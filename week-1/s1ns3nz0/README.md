# 1주차 과제: 고객 문의 티켓 구조화 추출

## 1. 사용한 모델, SDK, 실행 환경

| 항목 | 값 |
|------|-----|
| 모델 | `claude-sonnet-4-20250514` (Claude Sonnet 4) |
| SDK | `anthropic` Python SDK (v0.52+) |
| 검증 | `pydantic` v2 |
| Python | 3.12.11 (pyenv) |
| OS | macOS (Darwin 25.3.0) |

## 2. 실제 요청 구조

### system 메시지

- **v1**: 최소한의 지시. JSON 스키마를 보여주고 "JSON만 반환하라"고 지시
- **v2**: 한국 이커머스 전문가 역할 부여 + 각 필드별 한국어 정의 + 분류 규칙 4개 + 예시 1건 포함

### user 메시지

고객 문의 텍스트를 그대로 전달:
```
주문한 러닝화가 아직 도착하지 않았어요. 배송이 어디까지 왔는지 확인하고 싶습니다.
```

### 파라미터

| 파라미터 | 값 | 이유 |
|---------|-----|------|
| `model` | `claude-sonnet-4-20250514` | 분류 작업에 충분한 성능, 비용 효율적 |
| `temperature` | `0.0` | 분류 작업이므로 재현성과 일관성 최우선 |
| `max_tokens` | `256` | JSON 출력이 100토큰 미만이므로 여유있게 설정 |

## 3. 실제 응답 구조

### JSON 본문 예시 (ticket-02)

```json
{
  "intent": "payment_issue",
  "urgency": "high",
  "needs_clarification": false,
  "route_to": "billing_ops"
}
```

### Token 사용량

| 버전 | Input Tokens | Output Tokens | 합계 |
|------|-------------|--------------|------|
| v1 | 2,027 | 561 | 2,588 |
| v2 | 8,867 | 429 | 9,296 |

v2는 상세 프롬프트로 인해 input 토큰이 약 4.4배 증가했으나, output은 오히려 줄었다 (JSON만 깔끔하게 반환).

## 4. v1 → v2에서 바꾼 점

| 항목 | v1 | v2 |
|------|-----|-----|
| 역할 설정 | 일반 분류기 | 한국 이커머스 전문가 |
| 필드 정의 | 영문 enum만 나열 | 한국어 상세 설명 포함 |
| 분류 규칙 | 없음 | 4개 명시적 규칙 추가 (금전이슈→high, 배송완료 오표시→high 등) |
| 예시 | 없음 | 1-shot 예시 포함 |
| 출력 지시 | "Respond with ONLY the JSON" | "No explanation, no markdown fences" 추가 |

**핵심 변경 의도**: v1에서 urgency 판단이 모호한 경계 케이스(배송 완료 오표시, 미처리 불만)를 명시적 규칙으로 가이드.

### 대표 실패 3건과 원인

#### 1. ticket-06 (v1 실패, v2 성공)
- **입력**: "상품은 아직 받지 못했는데 앱에는 배송 완료로 떠 있습니다"
- **v1 예측**: urgency=`medium` / **정답**: urgency=`high`
- **v2 예측**: urgency=`high` (정답)
- **원인**: v1에는 "배송 완료 오표시는 high"라는 규칙이 없어 모델이 medium으로 판단. v2에서 명시적 규칙 추가로 해결.

#### 2. ticket-09 (v1, v2 모두 실패)
- **입력**: "지난주에 요청드린 건이 아직도 처리되지 않은 것 같아요"
- **예측**: urgency=`medium` / **정답**: urgency=`high`
- **원인**: "지난주에 요청한 건"이 무엇인지 맥락이 없어 모델이 긴급도를 보수적으로 판단. 정답은 고객 불만 고조를 반영해 high로 설정되어 있으나, 텍스트만으로는 판단이 모호한 경계 케이스.

#### 3. ticket-12 (v1, v2 모두 실패)
- **입력**: "사이즈가 안 맞아서 교환할지 환불할지 고민 중인데 어떤 절차로 진행하면 될까요?"
- **v1 예측**: urgency=`low`, needs_clarification=`true`
- **v2 예측**: urgency=`low`, needs_clarification=`false`
- **정답**: urgency=`medium`, needs_clarification=`true`
- **원인**: urgency — 모델이 "고민 중"이라는 표현을 긴급하지 않다고 해석. 정답 기준은 "처리가 필요한 건"이므로 medium. v2에서는 needs_clarification까지 틀림 — 상세 규칙이 오히려 "교환/환불 의도가 명확하다"고 과잉 판단하게 만듦.

### 분석 종합

- **intent**는 v1, v2 모두 100%로 가장 쉬운 필드. 한국어 문맥에서 의도 분류는 모델이 잘 수행.
- **urgency**가 가장 어려운 필드. 대부분의 실패가 urgency 판단에서 발생 (low↔medium↔high 경계).
- v2의 명시적 규칙이 **ticket-06 (배송 완료 오표시→high)**은 해결했으나, **ticket-11 route_to**를 오히려 v1이 더 잘 맞춤 — 규칙이 항상 긍정적 효과만 주지는 않음.
- **비용 관점**: v2는 input 토큰이 4.4배 많지만 exact match는 동일. 비용 최적화를 위해서는 v1 기반에서 urgency 판단 규칙만 추가하는 v3를 고려할 수 있음.
