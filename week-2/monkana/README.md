1. 프롬프트 기법 4가지 정리
Zero-shot Prompting
- 정의: 모델에게 예시를 주지 않고 오직 명령어만 전달하여 결과를 얻는 방식
- 핵심 원리
    - 방대한 사전 학습: 지식 및 대화/문맥의 구조, 단어 패턴 파악
    - 명령어 튜닝: 지시-답변 쌍 학습을 통한 의도 파악
    - 인간 피드백 기반 강화 학습: 모델이 내놓은 여러 답안 중 사람이 점수를 매겨 출력을 사람이 원하는 방식으로 정교하고 일관성있게 만들어감 
- 사용 예시: "다음 문장을 보고 기분을 한 단어로 맞춰봐. 문장: '오늘 친구랑 싸워서 선생님한테 혼났어'" -> 결과: "우울" 
- 언제 쓰면 좋은지
    - 간단하고 대중적인 작업: 감정 분석, 단순 요약, 번역 등 모델이 이미 잘 알고 있는 보편적인 일을 시킬 때.
    - 속도가 중요할 때: 예시를 길게 쓸 필요가 없어 프롬프트가 짧아지고 답변이 빠릅니다. (토큰 절약)
- 언제 효과가 적은지 
    - 특수/전문 분야: 모델이 학습 데이터에서 본 적 없는 '우리 회사 내부 매뉴얼'이나 '아주 최신 논문' 내용을 물어볼 때. (지식 자체가 없음)
    - 복잡한 논리 구조: "A이면 B이고, B가 아니면 C를 하되 D는 제외해" 같은 복잡한 조건문이 들어갈 때. (말귀는 알아먹어도 실수를 자주 함)
    - 엄격한 형식 요구: "반드시 3줄로 요약하고 끝은 '끝'으로 끝내" 같은 아주 세세한 출력 형식을 요구할 때.

Few-shot Prompting
- 정의: 모델에게 작업을 시키기 전에, 몇 개의 예시를 미리 보여주어 작업의 패턴과 형식을 학습시키는 방식
- 핵심 원리
    - 문맥 내 학습 (In-context Learning): 모델의 내부 파라미터를 수정(학습)하는 것이 아니라, 입력된 프롬프트 안에서 실시간으로 정답의 패턴과 형식을 파악합니다.
    - 조건부 확률 제어: 예시를 통해 특정 단어나 형식이 나올 확률을 높여, 모델이 사용자의 의도에 맞는 '레일' 위에서 답변하게 만듭니다.
    - 패턴 매칭 (Pattern Matching): "질문 -> 정답"의 구조를 반복해서 보여줌으로써, 모델이 마지막 질문에 대해서도 동일한 구조의 답변을 내놓게 강제.
- 사용 예시: 단순히 기분을 맞히는 게 아니라, [기분: 이모지] 형태로 답을 받고 싶을 때 유용
        - 문장: "와! 오늘 날씨 진짜 좋다!" -> 기분: 행복 😊
        - 문장: "지갑을 잃어버렸어..." -> 기분: 슬픔 😢
        - 문장: "내일 시험이라 너무 떨려." -> 기분:
            - 결과: "긴장 😰" (앞의 예시 패턴을 보고 단어와 이모지를 세트로 출력함
- 언제 쓰면 좋은지 
    - 까다로운 형식이 필요할 때: 정해진 양식(데이터 추출, 코드 생성 등)으로만 답을 들어야 할 때 
    - 제로샷으로 실패할 때: 모델이 자꾸 헛소리를 하거나 의도를 못 알아먹을 때 예시 2~3개만 줘도 성능이 확 올라감
    - 생소한 작업: 모델이 학습 데이터에서 자주 못 봤을 법한 아주 특이한 분류 작업을 시킬 때 유리
- 언제 효과가 적은지
    - 다단계 추론이 필요한 문제: 예시를 단순히 '질문-정답' 형태로만 주면, 수학 문제나 복잡한 논리 퍼즐처럼 풀이 과정이 중요한 작업에서는 여전히 오답을 낼 확률이 높습니다. (이때 CoT가 필요합니다.)
    - 예시 간의 모순: 제공된 예시들의 형식이 제각각이거나 논리가 서로 충돌하면 모델이 혼란에 빠져 성능이 급격히 저하됩니다.
    - 지식의 부재: 예시를 아무리 줘도 모델이 아예 모르는 전문 지식이나 최신 정보를 꾸며내야 하는 상황(Hallucination)은 해결하기 어렵습니다.
    - 긴 입력 데이터: 예시 자체의 길이가 너무 길면 정작 중요한 지시사항이 뒤로 밀려나거나, 모델이 처리할 수 있는 최대 길이(Token Limit)를 초과하여 뒷부분이 잘릴 수 있습니다.

Chain-of-Thought (CoT) Prompting
- 정의: 복잡한 추론 작업에 수행에 있어 최종 답을 내놓기 전에 중간 추론 단계를 쓰도록 유도하는 기법
- 핵심 원리
        단계적 분해 (Decomposition): 다단계가 필요한 복잡한 문제를 작은 단위의 논리적 단계로 쪼개어 해결합니다.
        중간 추론의 활용: 이전 단계에서 도출된 논리를 다음 단계의 근거로 사용함으로써, 논리적 비약이나 계산 실수를 줄입니다.
        충분한 모델 크기 필요: 이 기법은 주로 매개변수가 많은 대규모 언어 모델(LLM)에서 효과가 나타나며, 모델이 스스로 논리적 흐름을 구축할 수 있는 능력이 있어야 합니다.
- 사용 예시 (Usage Examples)
        - Few-shot CoT (예시를 통한 유도)
            단순히 "질문-정답" 쌍이 아니라, "질문 - [풀이 과정] - 정답" 형태의 예시를 모델에게 보여주는 방식입니다.
            프롬프트 예시:
                질문: 사과가 5개 있었는데 2개를 먹었어. 그 후 3개를 더 샀어. 몇 개일까?
                답변: 처음 5개에서 2개를 먹으면 3개가 남고, 여기에 3개를 더하면 6개가 돼. 정답은 6개야.
                (이후 실제 질문 투입)
        - Zero-shot CoT (명령어를 통한 유도)
            예시를 일일이 주기 힘들 때, 특정 문구 하나로 모델의 논리 회로를 깨우는 방식입니다.
                핵심 문구: "차근차근 단계별로 생각해보자 (Let's think step by step)"
                이 문장 하나만으로도 모델은 답변을 내놓기 전 스스로 내부적인 추론 과정을 생성하기 시작합니다.

- 언제 쓰면 좋은지
    - 산술 추론: 수학 문장제 문제 해결
    - 상식 추론: 원인과 결과, 물리적 현상에 대한 이해가 필요한 문제
    - 상징적 추론: 특정 규칙에 따라 글자를 조합하거나 논리 퍼즐을 푸는 작업
- 언제 효과가 적은지
    - 단순 지식 인출: 수도나 번역 같은 단순 사실 확인은 논리가 필요 없어 문장만 길어집니다.
    - 직관적/패턴 작업: 요약, 번역, 말투 변환 등 '추론'보다 '감각'이 중요한 작업엔 효과가 미미합니다.
    - 학습되지 않은 전문 지식: 모르는 지식을 논리로 메울 순 없기에, 아는 척하며 거짓말(환각)할 확률만 높아집니다.
    - 모델 체급: 소형 모델에서는 오히려 성능이 떨어지거나 엉뚱한 추론 과정을 만들어낼 수 있습니다.
    - 오류 누적: 중간 단계에서 한 번이라도 틀리면 최종 정답까지 도미노처럼 틀리게 됩니다.
    - 비용 발생: 추론 과정만큼 출력 토큰이 늘어나므로 속도가 느려지고 비용이 증가합니다.


Self-Consistency (자기 일관성)
- 정의 (Definition): 모델에게 하나의 질문에 대해 여러 개의 서로 다른 추론 경로를 만들게 한 뒤, 그 중 많이 선택한 답을 다수결 원칙에 따라 최종 답으로 선정하는 방식
- 핵심 원리 (Core Principles):
    - 다양한 샘플링 (Sampling Diverse Paths): 모델이 매번 똑같은 대답만 하는 게 아니라, 여러 갈래의 생각(CoT)을 해보도록 유도합니다.
    - 일관성 확인 (Consistency Checking): 여러 번의 독립적인 추론 시도 중 가장 공통적으로 나타나는 답이 정답일 확률이 높다는 통계적 원리를 이용합니다.
    - 우연한 오류 제거: 단일 시도에서 발생할 수 있는 사소한 계산 실수나 논리적 도약을 다른 올바른 추론 결과들로 보완(상쇄)합니다.

- 사용 예시 (Usage Example):
    - 방식: 복잡한 수학이나 논리 문제를 줄 때, "이 문제를 5가지 서로 다른 방식으로 생각해서 풀고, 가장 많이 나온 공통적인 최종 답을 알려줘"라고 지시합니다.
    - 결과: 모델이 '풀이 A -> 답 10', '풀이 B -> 답 10', '풀이 C -> 답 5' 등을 내놓으면, 다수결에 따라 가장 일관성 있는 '10'을 채택합니다.
- 언제 쓰면 좋은지 (Pros)
    - "답이 틀리면 큰일 나는 상황" (정확도가 생명일 때)
    - "풀이 과정이 여러 개일 수 있는 문제" (다양한 각도에서 검토가 필요할 때)
    - "CoT를 썼는데도 자꾸 계산 실수가 나올 때" (모델의 '운' 요소를 제거하고 싶을때)
- 언제 효과가 적은지 (한계점):
    - 정답이 없는 창의적 작업: 에세이 쓰기나 시 창작처럼 '다수결'로 정답을 정할 수 없는 작업에는 의미가 없습니다.
    - 모델의 근본적 지식 부족: 모델이 애초에 잘못된 사실을 진실로 믿고 있다면, 100번을 물어도 똑같이 틀린 답만 나오기 때문에 효과가 없습니다.
    - 단순 사실 확인: "미국의 수도는?" 같은 질문은 추론 경로가 생길 여지가 없으므로, 여러 번 시도하는 것 자체가 시간과 비용 낭비입니다.
    - 고비용 저효율: 동일한 질문을 여러 번 생성해야 하므로 사용료(토큰)가 많이 들고 응답 속도가 현저히 느려집니다.
