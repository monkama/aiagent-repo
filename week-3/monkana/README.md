1. RAG란
- Retrieval-Augmented Generation의 약자
- 말 그대로 검색(Retrival)으로 바깥 지식을 가져와서 그걸 바탕으로 생성하는 방식
- 내부에 있는 지식만으로는 정확하고 최신화된 답변을 내기 힘들기에 해당 모델이 만들어졌다.


dense vector: 문장이나 문서를 의미를 기반으로 숫자배열로 바꾼 것이고,
index: 그 백터들을 빠르게 찾기위해 정리해둔 검색 구조이며,
dense vector index: 문서들을 의미 기반 백터로 저장하고 질문과 비슷한 문서를 빠르게 찾게 해주는 인덱스이다.

2. system prompt에 전체 데이터를 삽입하는 방식과 rag 차이
전체 데이터 삽입 같은 경우, 내가 주는 데이터만으로 판단이 이뤄지기에 최신화된 답볍이 어려울 수 있고, 필요한 데이터 량이 많을때는 프롬프트 입력 용량 한계로 인해 힘들다.
그러나 rag 같은 경우, 참고할 수 있는 데이터 셋을 백터 DB에 저장하고, 사용자 질문과 연관성이 있는 부분만 찾아와서 LLM에게 질문과 함께 전달하기에 훨씬 더 많은 데이터 셋 안에서 LLM이 판단할 기회를 줄 수 있다.

3. RAG 파이프라인의 전체 흐름
- Indexing: 문서를 검색 가능한 형태로 미리 준비하는 단계
- Retrieval + Generation: 사용자의 질문이 들어왔을 때 관련 내용을 찾아 답을 만드는 단계

Indexing 단계: 문서를 로드하여 청크 단위로 하여 작게 분할한 뒤, 각 청크를 숫자 백터로 임베팅 한다음, 백터 스토어에 저장
Retrieval + Generation: 사용자가 질의 하면, 질의를 청크와 같이 백터로 바꾼 다음, 질문 백터와 저장된 백터들을 비교하여 가장 비슷한 청크 Top-K를 찾고, 찾아온 청크들을 LLM에게 전달하여 답 생성

3. RAG에서 사용되는 용어
Chunking  긴 문서를 검색 가능한 작은 단위로 나누는 것
문서를 문단,문장,고정 길이 단위로 잘라낸 텍스트 조각를을 청크라고 하며, 해당 청크를 임배딩하여 백터로 저장한다 
이때, 방금 말한 청크의 단위 크기를 Chunk_size이며, 만약 청크의 텍스트 정보가 잘리는 상황을 방지하기위해 일부러 청크가 서로 겹쳐 자를 수 있게 하는 설정을 chunk_overlap이라 함.

Embeding 청크를 의미 기반 백터로 변환하는 것
벡터: 부동소수점 숫자 리스트
벡터간 거리가 작을수록 관련성이 높고, 검색 군집화 추천 분류 같은 작업에 활용

Vector Store 백터를 저장하고 유사한 백터를 빠르게 찾기위한 저장 검색 구조
많은 백터를 빠르게 검색하기위해 백터 DB 사용 권장
백터들을 데이터베이스 벡터 집합으로 두고, 질의 벡터에 대해 최근접 이웃을 모색


Retriever: 질문 벡터와 가장 가까운 청크 벡터를 찾아오는 검색 모듈
백터 DB의 청크 백터와 질의 백터를 비교해 가장 가까운 결과를 가져오고, 이때 몇 개까지 가장 가까운 결과를 가져올지를 정하는게 Top-K 라고 한다. 

Generation: Top-K 청크 원문을 질의와 함께 LLM 프롬프트에 넣고, 모델이 이를 바탕으로 답변 생성

1. 질의 전체 의미가 전자 청크와 더 비슷
2. 청크 내부에 다른 단어들이 더 강하게 작용
3. 청킹 방식 때문에 의미가 섞였거나
4. 후자 청크가 오히려 노이즈가 많을 수 있거나
5. 질문 문장 자체가 후자 청크를 강하게 가리키지 않았거나
6. 순수 벡터 검색의 한계

위 문제들을 요약하자면, 결국 청킹시 의미 단위로 청킹해줘야 백터 검색의 효율을 극대화할 수 있는 것으로 판단된다. 그래서 pdf 청킹과 관련된 전문 라이브러리를 찾아본 결과, Unstructured, Docling, PyMuPDF4LLM, Marker 등이 있다. 
Unstructured: PDF를 바로 텍스트 덩어리로 보지 않고, Title, NarrativeText, ListItem 같은 문서 요소 단위로 분해하는 게 강점인 라이브러리다. 보고서처럼 제목-본문-목록-표 구조를 갖고있으면 가장 유용한 툴이기에 이번 과제에 해당 툴을 이용하는게 가장 유용해보여 추후 과제 수정시 사용할 예정이다.
Docling: 계층형 chunking 쪽에 강하다. 특정 청킹이 어느 계층에 속한 것인지에 강한 툴이기에 발표자료, 보고서, 기술문서, 연구자료 쪽에서 특히 좋다고 나온다.
PyMuPDF4LLM: 구조화된 Markdown으로 뽑는 것을 전문으로 한다. 
Marker: PDF를 Markdown, JSON, chunks, HTML로 변환하는 데 초점을 둔다. PDF를 LLM 친화적인 중간 표현으로 바꾸는 전처리기로 상당히 강하다. 전 2주차 과제인 프롬프트 과제에서는 구조화된 입력 데이터를 넣는것이 핵심이었는데, 해당 툴을 활용해보는 것이 좋다고 본다. 추후 2주차 과제도 수정시 해당 툴을 사용해볼 예정이다.

그리고 PDF에서 추출된 표 텍스트가 일부 깨져 있었지만, 핵심 항목명과 수치 정보가 동일 청크 내에 포함되어 있어 의미 기반 검색 및 간단한 계산 등은 LLM 추론을 통해 관련 정보를 비교적 정확히 찾을 수 있었다. 다만 이는 표 구조가 완전하게 보존된 것은 아니므로, 정밀한 서비스 단계에서는 표 구조화 전처리가 추가로 필요하다고 본다. 관련된 파이썬 라이브러리를 찾아본 결과, Camelot, tabula-py, pdfplumber, Docling이 있다. 이 또한 필요한 작업이라 판단되어 추후 과제 수정시 사용해볼 예정이다.
