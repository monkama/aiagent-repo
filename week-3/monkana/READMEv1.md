## 1. RAG란

- **Retrieval-Augmented Generation**의 약자
- 검색(Retrieval)으로 바깥 지식을 가져와서 그걸 바탕으로 생성하는 방식
- 내부에 있는 지식만으로는 정확하고 최신화된 답변을 내기 힘들기에 해당 모델이 만들어졌다

| 용어 | 설명 |
|------|------|
| dense vector | 문장이나 문서를 의미를 기반으로 숫자배열로 바꾼 것 |
| index | 그 벡터들을 빠르게 찾기 위해 정리해둔 검색 구조 |
| dense vector index | 문서들을 의미 기반 벡터로 저장하고 질문과 비슷한 문서를 빠르게 찾게 해주는 인덱스 |

## 2. system prompt에 전체 데이터를 삽입하는 방식과 RAG 차이

**전체 데이터 삽입 방식**
- 내가 주는 데이터만으로 판단이 이뤄지기에 최신화된 답변이 어려울 수 있음
- 필요한 데이터 량이 많을 때는 프롬프트 입력 용량 한계로 인해 힘듦

**RAG 방식**
- 참고할 수 있는 데이터셋을 벡터 DB에 저장하고, 사용자 질문과 연관성이 있는 부분만 찾아와서 LLM에게 질문과 함께 전달
- 훨씬 더 많은 데이터셋 안에서 LLM이 판단할 기회를 줄 수 있음

## 3. RAG 파이프라인 전체 흐름

### Indexing 단계
문서를 검색 가능한 형태로 미리 준비하는 단계

```
문서 로드 → 청크 단위로 분할 → 각 청크를 숫자 벡터로 임베딩 → 벡터 스토어에 저장
```

### Retrieval + Generation 단계
사용자의 질문이 들어왔을 때 관련 내용을 찾아 답을 만드는 단계

```
사용자 질의 → 질의를 벡터로 변환 → 저장된 벡터들과 비교 → Top-K 청크 추출 → LLM에 전달 → 답변 생성
```

---

## 4. RAG에서 사용되는 용어

| 용어 | 설명 |
|------|------|
| **Chunking** | 긴 문서를 검색 가능한 작은 단위로 나누는 것. 문단·문장·고정 길이 단위로 자른 텍스트 조각을 청크라고 함 |
| **Chunk_size** | 청크의 단위 크기 |
| **Chunk_overlap** | 청크의 텍스트 정보가 잘리는 상황을 방지하기 위해 청크가 서로 겹쳐 잘리도록 하는 설정 |
| **Embedding** | 청크를 의미 기반 벡터로 변환하는 것. 벡터 간 거리가 작을수록 관련성이 높음 |
| **Vector Store** | 벡터를 저장하고 유사한 벡터를 빠르게 찾기 위한 저장·검색 구조 |
| **Retriever** | 질문 벡터와 가장 가까운 청크 벡터를 찾아오는 검색 모듈. 몇 개까지 가져올지를 정하는 게 **Top-K** |
| **Generation** | Top-K 청크 원문을 질의와 함께 LLM 프롬프트에 넣고, 모델이 이를 바탕으로 답변 생성 |

## 5. 의료급여 PDF 청킹 시 예측한 어려움

> 청크 단위로 청킹 시, 표처럼 같이 있어야 하는 묶음들이 서로 다른 청크가 될 수 있다는 어려움이 있을 것 같다.

## 7. 검색 품질 확인 결과

| 질문 ID | 난이도 | 검색 결과 | 예상 답변 | 모델 답변 | 답변 일치 |
|--------|--------|:-------:|---------|---------|:-------:|
| q01 | easy | 성공 | 20% | 20% | O |
| q02 | easy | 성공 | 무료 | 무료 | O |
| q03 | hard | 성공 | 해당되지 않음 | 해당되지 않음 | O |
| q04 | medium | 성공 | 60,000원 | 60,000원 | O |
| q05 | medium | **실패** | 45,000원 | 45,000원 | O |

**검색 성공률: 4/5**

### 검색 실패 원인 분석 (q05)

q05에서 검색 시 필수로 포함되어야 할 단어인 `조현병`이 포함되지 않은 것으로 보였다. 그럼에도 계산이 맞은 이유를 확인해보니, 9페이지 `의료급여기관 이용 시 본인부담률 및 부담액` 항목 중 `CT, MRI, PET 등` 때문에 15%라는 본인부담률을 보고 계산은 맞았으나 정확한 검색은 아닌 것으로 확인된다.

원인 요약:
1. 질의 전체 의미가 다른 청크와 더 비슷
2. 청크 내부에 다른 단어들이 더 강하게 작용
3. 청킹 방식 때문에 의미가 섞였거나
4. 후자 청크가 오히려 노이즈가 많을 수 있거나
5. 질문 문장 자체가 후자 청크를 강하게 가리키지 않았거나
6. 순수 벡터 검색의 한계

실제로 검색된 청크를 확인해보면 문제가 명확해진다.

```
[의료급여기관 이용 시 본인부담률 및 부담액] 청크 2
청크 인덱스: 6  |  page: 4 (page_label: 5)
------------------------------------------------------------
• 응급환자
• 장애인 보조기기를 지급받는 경우

의료급여 본인일부부담금3
의료급여기관 이용 시 본인부담률 및 부담액
입원 식대 본인부담률
2, 3인실 입원료 본인부담률
01  02  03

구분       1차(의원)  2차(병원,종합병원)  3차(상급종합병원)  약국   CT,MRI,PET등
1종 입원   무 료      무 료              무 료             -     무 료
1종 외래   1,000원   1,500원            2,000원           500원  5%
2종 입원   10%       10%               10%               -     10%
2종 외래   1,000원   15%               15%               500원  15%

구분                              본인부담률
1종·2종 / 2종장애인                20%
중증질환(합병증 포함)               5%
행려환자, 자연분만, 6세 미만 아동    무 료

구분               2인실  3인실
상급종합병원         50%   40%
종합병원·한방병원 등  40%   30%

※ 경증질환 약국 약제비 본인부담률 3% 적용
※ 비급여는 전액본인부담, 선별급여는 항목별로 30~90% 본인부담
※ 본인부담 보상제·상한제 해당되지 않음
```

위 청크 하나에 `의료급여기관 이용 시 본인부담률`, `입원 식대 본인부담률`, `2·3인실 입원료 본인부담률` 세 가지 주제가 뒤섞여 있는 것을 확인할 수 있다. 여러 의미가 섞인 채로 청크가 생성되는 바람에 벡터 오염이 발생하였고, 그로 인해 사실상 필요한 내용과 필요 없는 내용까지 함께 검색되는 사태가 발생하였다. 검색 결과는 성공으로 뜨나 큰 의미가 없는 이유가 바로 이 때문이다.

> 결국 청킹 시 의미 단위로 청킹해줘야 벡터 검색의 효율을 극대화할 수 있으며, 이는 추후 반드시 해결해야 할 숙제로 남는다.

## 8. Golden Dataset 5문제 중 검색 실패 예측

> hard 문제에서 검색 실패가 많을 것 같다. hard 문제 중 예외 사항이 있는데 이 예외 사항이 만약 한 청크 단위에 있지 않고, 서로 다른 청크에 존재할 경우, Retrieval 과정에서 정확히 찾지 못할 가능성이 존재할 것 같다.

## 9. 향후 개선 방향

### PDF 청킹 전문 라이브러리

| 라이브러리 | 특징 |
|-----------|------|
| **Unstructured** | PDF를 Title, NarrativeText, ListItem 같은 문서 요소 단위로 분해. 보고서처럼 제목-본문-목록-표 구조에 유용 |
| **Docling** | 계층형 chunking에 강함. 발표자료, 보고서, 기술문서, 연구자료 쪽에 특히 좋음 |
| **PyMuPDF4LLM** | 구조화된 Markdown으로 뽑는 것을 전문으로 함 |
| **Marker** | PDF를 Markdown, JSON, chunks, HTML로 변환하는 데 초점. LLM 친화적인 중간 표현으로 바꾸는 전처리기 |

### 표 구조화 전처리 라이브러리

PDF에서 추출된 표 텍스트가 일부 깨져 있었지만, 핵심 항목명과 수치 정보가 동일 청크 내에 포함되어 있어 의미 기반 검색 및 간단한 계산은 LLM 추론을 통해 비교적 정확히 처리할 수 있었다. 다만 정밀한 서비스 단계에서는 표 구조화 전처리가 추가로 필요하다.

관련 라이브러리: **Camelot**, **tabula-py**, **pdfplumber**, **Docling**
