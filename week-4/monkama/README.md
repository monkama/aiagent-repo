# 4주차 실습: Basic RAG + Advanced RAG (Hybrid Search & Re-ranking)

## 이론 정리

### 1. Hybrid Search란?

**정의:** 벡터 검색 방식과 키워드 검색 방식을 결합한 검색 방식

**벡터 검색만으로 부족한 이유**
의미적으로 비슷한 문서를 잘 찾지만, 사용자 질의에 포함된 특정 토큰을 찾는다고 보장해주지 않는다. 특히 특정 도메인에 대한 학습이 부족한 모델에게 맡길 경우, 해당 도메인 토큰을 놓칠 수 있다.

**BM25 알고리즘의 핵심 원리 (TF-IDF 기반, 정확한 토큰 매칭)**
자주 등장하는 토큰에는 낮은 점수를, 흔하지 않은 토큰에는 높은 점수를 부여한다. 토큰 매칭은 정확한 매칭 기반이며, 문서 내 빈도·전체 문서 집합에서의 희소성·문서 길이를 함께 반영하여 각 문서에 점수를 매기고, 높은 점수 순으로 상위 랭킹에 노출시킨다.

**벡터 검색 vs BM25 장단점 비교**

| 검색 방식 | 강점 | 약점 |
|----------|------|------|
| 벡터 검색 | 정확한 키워드 매칭이 아니더라도 질의의 의미를 파악하여 적절한 문서를 찾는다 | 도메인 특화 용어 등 특정 토큰을 중요한 상황에서 놓칠 가능성이 있다 |
| BM25 | 도메인 특화 용어 등 누락시키지 말아야 할 토큰에 강하다 | 의미가 다른 문서가 상위에 노출될 수 있으며, 동의어·유사어·바꿔 말하기 상황에 약하다 |
| Hybrid (결합) | 의미 파악과 정확한 키워드를 함께 반영하여 도메인 특화 용어가 중요한 질의에 더 정확한 문서를 찾을 가능성이 높다 | Dense·Sparse 비중을 조절하는 튜닝이 필요하며, 구현 방식에 따라 운영 복잡도가 증가할 수 있다 |

---

### 2. Re-ranking이란?

**정의:** 사용자 질의에 더 정확한 문서를 제공하기 위해, 검색된 문서 후보들의 순위를 특정 모듈로 재정렬하는 방식

**RAG 파이프라인에서의 위치**
Retrieval로 나온 Top-k 후보군의 순위를 재정렬한 뒤 LLM에 전달하므로, RAG 파이프라인에서 Retrieval과 Generation 사이에 위치한다.

**Bi-encoder vs Cross-encoder 비교**

| 구분 | Bi-encoder | Cross-encoder |
|------|-----------|---------------|
| 입력 방식 | 질문과 문서를 각각 인코딩 | 질문과 문서를 함께 인코딩 |
| 속도 | 문서를 미리 임베딩해두고 질의만 따로 임베딩 후 코사인 유사도를 비교하므로 빠르다 | 질문-문서 쌍을 모델에 하나씩 넣어 점수를 내야 하므로 느리다 |
| 정확도 | 질문과 문서를 각각 인코딩하여 단어 직접 비교가 불가능하므로 상대적으로 낮다 | 질문과 문서를 함께 인코딩하여 단어 직접 비교 및 문맥 파악이 정교하므로 높다 |

→ 보통 Bi-encoder로 상위 후보군을 먼저 추린 뒤, 해당 후보군에 Cross-encoder 방식으로 재정렬(Re-ranking)하는 구조를 사용한다.

**코사인 유사도 함수**
- 두 벡터가 얼마나 같은 방향을 바라보는지 확인하는 함수: `내적 / 두 벡터 크기의 곱`
- 내적: 두 벡터의 방향 관계와 크기를 함께 반영한 값

**상용 Re-ranking API: Cohere Rerank**
Cross-encoder 방식과 유사하게 질의-문서를 맞대어 비교한 뒤 `relevance_score`를 매겨 순위를 재정렬한다. 100개 이상 언어를 지원하며, YAML 문자열 포맷으로 문서를 전달하는 것을 권장한다.

### 3. Metadata Filtering과 Context Compression

**Metadata Filtering**
청크에 포함된 메타데이터를 기반으로 검색 범위를 제한하는 방법. Retrieval 시 의미적 유사성에 의존하지 않고 제한된 범위 내에서 검색이 이뤄지므로 신뢰성 있는 검색이 가능하다. 적용하지 않을 경우, 같은 내용을 다루는 연도가 다른 문서들 사이에서 의미적으로 유사한 과거 연도 문서를 참조하는 문제가 발생할 수 있다.

**Context Compression**
검색으로 가져온 청크 전체를 그대로 LLM에 전달하지 않고, 질문에 필요한 내용만 남기도록 압축하는 방법. 질의와 직접 관련된 문장·핵심 정보만 추려내어 최종 컨텍스트로 사용하므로, 불필요한 정보가 줄어들어 LLM이 핵심 정보에 집중할 수 있다. 입력 토큰도 줄어 전체 토큰 사용량을 절감할 수 있다.

### 4. 실습 전 가설 및 실습 결과

**Q. 2025년과 2026년 문서를 동시에 인덱싱했을 때, 년도 혼동이 가장 많이 발생할 질문 유형은?**

> 가설: 두 연도를 같이 확인해야 하는 cross-year 문제에서 혼동이 많이 발생할 것이다. 같은 내용인데 연도만 다르기 때문에 임베딩 시 거의 같은 벡터 방향으로 향하기 때문이다. 메타데이터에 `source_year`를 기입하여 메타데이터 필터링이 핵심이라 본다.

> 실습 결과: 가설이 맞았다. cross-year 문항이 가장 낮은 정답률(Basic RAG 기준 71%)을 보였다. 다만 메타데이터 필터링을 적용한 결과, 전 구간에서 년도 검색 정확도 25/25(100%)를 달성했다. 즉 년도 혼동 자체는 메타데이터 필터링으로 완전히 해소됐고, 남은 오답의 원인은 LLM 해석 오류와 검색 실패였다.

---

**Q. Hybrid Search가 가장 효과를 볼 질문 유형은?**

> 가설: 특정 의료 용어가 포함된 질문 유형에서 효과를 볼 것이다. 의미 기반 탐색만으로는 도메인 특화 용어를 놓칠 가능성이 있기 때문이다.

> 실습 결과: 가설과 부분적으로 일치했다. BM25가 실제로 cross-year 문항(q21: 디스크·협착증·복잡추나 등 도메인 용어)에서 검색 보완 효과를 보였다. 다만 BM25 인덱스에 table 데이터까지 포함시켰을 때는 오히려 노이즈로 작용해 Basic RAG보다 낮아지는 역효과가 발생했다. BM25 인덱스를 subject+text만으로 제한하고 top-k를 조정했을 때 최종 92%로 Basic RAG(88%)를 초과했다.

---

**Q. Re-ranking이 결과를 개선할지, 오히려 악화시킬 수 있는 경우가 있을지?**

> 가설: 리랭킹 모델에게 전달하는 문서의 핵심 요약을 어떻게 만들어내는지가 관건일 것이다. 요약이 부실하면 낮은 점수가 부여되어 역효과가 날 수 있다.

> 실습 결과: 가설이 정확히 맞았다. Cohere Rerank에 subject만 전달했을 때 64%, subject+text 전달 시 80%로, 전달 데이터 품질에 따라 결과가 크게 달랐다. 그리고 어떤 조합을 써도 Rerank 없는 RRF(92%)보다 낮았다. RRF가 이미 잘 정렬한 순서를 Reranker가 오히려 망가뜨리는 패턴이 관찰됐다.

---

**Q. 메타데이터 필터링이 년도 혼동 문제를 얼마나 해결할 수 있을지?**

> 가설: 검색 범위가 한정되므로 속도·정확도 측면에서 큰 향상이 있을 것이다. 특히 연도별로 값이 다른 cross-year 문항에서 명확한 구분이 필요하다.

> 실습 결과: 예상보다 더 강력하게 작동했다. Basic RAG부터 Advanced RAG 전 구간에서 년도 검색 정확도 25/25(100%)를 달성했으며, 년도 혼동으로 인한 오답은 단 한 건도 발생하지 않았다. 남은 오답의 원인은 "올바른 연도의 청크를 가져오지 못한 것"이 아니라, 올바른 연도 범위 안에서 검색했음에도 정답 근거가 되는 특정 키워드(예: 장기지속형 주사제, 복잡추나 등)가 포함된 청크 자체를 Top-K 안에 가져오지 못한 검색 실패, 또는 청크는 올바르게 검색됐지만 LLM이 컨텍스트를 잘못 해석한 오류에 해당했다.

### 인덱싱 전략

**PDF 파싱 및 청킹 전략**

초기에는 `pdfplumber`로 전체 문서를 파싱하고, Upstage Document Parse가 추출한 표 markdown만 별도로 가져와서 `page + heading` 문자열 매칭으로 기존 pdfplumber 청크에 끼워 넣는 방식을 사용하였다. 하지만 실제 문서에서는 Upstage의 `heading`, `paragraph`, `table` 구분이 일관적이지 않았고, 표 제목이나 각주가 heading/paragraph로 분리되거나 표 내부에 섞이는 경우가 많았다. 이로 인해 표가 잘못된 청크에 붙거나, 필요한 표가 누락되는 문제가 발생하였다.

이를 해결하기 위해 최종 파이프라인에서는 pdfplumber와 Upstage의 역할을 page 단위로 분리하였다. 먼저 `pdfplumber`로 표가 있는 페이지를 감지하고, 표가 없는 페이지는 기존처럼 pdfplumber 기반 텍스트를 사용하여 청킹하였다. 반면 표가 있는 Ⅰ. 의료급여제도 페이지는 pdfplumber 청크에 Upstage 표를 끼워 맞추지 않고, Upstage가 해당 페이지에서 추출한 elements를 직접 사용하여 우리가 정한 청크 기준에 따라 새로 청킹하였다. 이후 pdfplumber 기반 청크와 Upstage 기반 청크를 page 순서대로 병합하였다.

청크 단위는 문서 구조에 맞춰 정의하였다. Ⅰ. 의료급여제도는 대분류를 `parent_subject`, 세부 항목을 `subject`로 두었고, `03 의료급여 본인일부부담금`의 `9 의료급여 2종수급권자 본인부담률` 항목은 `○` 항목 단위로 다시 나누었다. Ⅱ. 자주하는 질문 영역은 Q/A 단위로 청킹하였다. 각 청크는 `source`, `section`, `parent_subject`, `subject`, `text`, `tables`, `page` 구조로 저장하였다.

표 처리 과정에서는 단순히 Upstage의 `category == table`만 신뢰하지 않았다. 실제 PDF에서는 표 제목이 table 첫 row에 섞이거나, 표 header인 `구분`이 paragraph/list로 따로 빠지거나, 일부 셀이 table 밖 paragraph로 분리되는 문제가 있었다. 또한 footer 문구나 페이지 번호가 table row 안에 섞이는 경우도 있었다. 따라서 `category == table`을 기본 표 시작 기준으로 두되, `구분` 또는 `구분 의료급여기관`처럼 짧은 paragraph/list가 table 직전에 나타나는 경우에는 이를 다음 table의 header로 간주하여 table에 붙였다.

표의 끝은 새 청크 시작 신호를 기준으로 판단하였다. 대분류 번호, 세부 항목 번호, `○` 항목, Q/A 경계가 나타나면 이전 표가 끝난 것으로 보았다. `※`로 시작하는 주석이 등장하는 경우에도 앞의 table은 종료된 것으로 처리하고, 이후 내용은 text/note로 분리하였다. 다만 Upstage가 표 내부 값을 paragraph로 떨어뜨리는 경우가 있어, 새 청크나 `※`가 나오기 전의 짧은 값은 table spillover로 간주하여 table에 복원하였다.

일부 표는 Upstage의 markdown/html 결과 모두에서 구조가 크게 깨져 자동 후처리만으로 복원하기 어려웠다. 특히 2025년 `의료급여기관 이용 시 본인부담률 및 부담액` 표와 `추나요법 본인부담률` 표는 병합 셀 구조가 깨지거나 일부 값이 paragraph로 분리되어 잘못된 구조로 저장되었다. 이 두 표는 RAG 평가에서 중요한 표였기 때문에 known broken table로 분류하고, 수동으로 정상 markdown 형태를 정의하여 교체하였다.

**청킹 구조**

```
section / parent_subject / subject / text / tables / page / source(연도)
```

**메타데이터 설정**

각 청크를 ChromaDB에 저장할 때 아래 메타데이터를 포함시켜 연도별 필터링이 가능하도록 했다.

```python
metadata = {
    "source_year": "2025" or "2026",   # 메타데이터 필터링 핵심
    "section": ...,
    "parent_subject": ...,
    "subject": ...,
    "page": ...,
    "text": ...,
    "tables": ...,    # JSON 직렬화
}
```

**page_content 구성 (LLM 전달용)**

```
[출처년도: {source_year}]
{subject}
{text}
{tables}
```

출처 연도를 page_content 앞에 명시하여 LLM이 연도를 직접 인식할 수 있도록 구성했다.

| 2026 | 9문항 (q02, q04, q06, q07, q10, q12, q14, q16, q17) |

**난이도별 구성**

| 난이도 | 문항 수 | 설명 |
|--------|---------|------|
| easy | 8문항 | 단일 조건으로 표에서 직접 조회 가능한 문항 |
| medium | 6문항 | 2~3개 조건을 복합적으로 적용해야 하는 문항 |
| hard | 4문항 | 조건 조합 후 계산까지 필요한 문항 |
| cross-year | 7문항 | 두 연도 정보를 동시에 검색·비교해야 하는 문항 |

**cross-year 문항 설계 의도**

두 연도 문서는 구조와 표현이 매우 유사하여 벡터 검색만으로는 연도를 구분하기 어렵다. cross-year 문항은 이 취약점을 집중적으로 테스트하기 위해 설계했다. 유형은 크게 세 가지다.

- **값이 달라진 경우**: q19(장기지속형 주사제 5%→2%), q20(조산아 지원 기간 확대)
- **값이 동일한 경우**: q21(복잡추나 40% 동일), q24(MRI 15% 동일) — 동일하다는 판단 자체를 테스트
- **2026년 신설 제도**: q23(연간 외래 365회 초과 30% 신설), q22(응급의료관리료 KTAS 기준 변경)

## Step 1. Basic RAG 완성 결과

### 파이프라인 구축 흐름

1. PDF 원문 처리
- 2025년, 2026년 의료급여제도 PDF를 각각 처리하였다.
- 문서 본문에 해당하는 5~35페이지만 사용하여 표지, 목차, 부록성 페이지가 섞이지 않도록 제한하였다.
- `pdfplumber`로 표가 있는 페이지를 감지하고, 표가 있는 페이지는 Upstage Document Parse 결과를 활용하였다.

2. 청킹 및 표 처리
- 표가 없는 페이지는 `pdfplumber` 기반 텍스트 추출후 청크 결과를 저장하였다.
- 표가 있는 Ⅰ. 의료급여제도 페이지는 Upstage가 추출후 청크 결과를 저장했다.
- 청크는 `section`, `parent_subject`, `subject`, `text`, `tables`, `page`, `source` 구조로 저장하였으며, 해당 주로 해당 key 값들을 기준으로 청킹하였다. 다만 예외적으로 2종 수급권자 본인부담율 같은 경우, 예외 기준을 적용하여 청킹하였는데, 원래는 subject 에 해당하는 청크의 text 형태로 저장되던 ○ 패턴을 해당 경우만 예외적으로 청크 단위로 취급해주었다.
- `text`와 `tables`를 분리하여 저장하고, RAG 입력 시 둘을 함께 컨텍스트로 전달하였다.
- footer, 페이지 번호, `Ⅰ. 의료급여제도 •` 같은 반복 노이즈는 제거하였다.
- `구분`이 paragraph로 분리된 표 헤더는 다음 table의 header로 보정하였다.
- 2025년 표 추출 품질이 깨진 일부 표는 예외적으로 수동 정규화하였다.
  - `의료급여기관 이용 시 본인부담률 및 부담액`
  - `추나요법 본인부담률`

3. 통합 청크 생성
- 2025년 청크: 91개
- 2026년 청크: 91개
- 통합 청크: 182개
- 각 청크에는 출처 연도인 `source` 값을 포함하였다.

4. 벡터DB 구축
- 통합 청크 `all_chunks(2025,2026).json`을 ChromaDB에 저장하였다.
- 청크 본문에는 출처년도, subject, text, tables를 함께 넣어 LLM이 출처 연도를 인식할 수 있도록 구성하였다.
- 메타데이터에는 `source_year`, `section`, `parent_subject`, `subject`, `page`를 저장하였다.

5. 검색 단계
- 질문에 `2025`, `2026`이 들어 있는지 확인하여 검색할 연도를 먼저 추론하였다.
- 단일 연도 질문은 Chroma 검색 전에 `source_year` 메타데이터 필터로 해당 연도 청크만 검색하였다.
- `2025,2026` cross-year 질문은 두 연도에서 각각 검색한 뒤 결과를 합쳐 LLM에게 전달하였다.

6. Generation 단계
- 검색된 Top-K 청크를 출처년도와 page 정보가 포함된 컨텍스트로 구성하였다.
- LLM에게 "질문이 특정 년도를 묻는 경우 해당 년도의 출처 컨텍스트만 사용하라"고 지시하였다.
- 컨텍스트에 없는 내용은 `정보를 찾을 수 없습니다`라고 답하도록 프롬프트를 구성하였다.

7. 평가 방식
- Golden Dataset 25문항으로 평가하였다.
- 일반 문항은 expected answer의 핵심 값이 LLM 답변에 포함되는지로 판정하였다.
- cross-year 문항은 표현 차이로 인한 오판정을 줄이기 위해 별도 LLM judge에게 `질문`, `모범정답`, `모델답변`을 전달하여 의미상 정답 여부를 판정하도록 하였다.
- 검색 성공 여부와 올바른 연도 검색 여부를 함께 기록하였다.

## Step 2. Advanced RAG (Hybrid Search + Rerank) 튜닝 과정 및 결과

### 구성

- **Dense**: ChromaDB + OpenAI `text-embedding-3-large`
- **Sparse**: BM25 (langchain-community BM25Retriever)
- **앙상블**: EnsembleRetriever (RRF 방식으로 두 결과 병합)
- **Reranker**: Cohere `rerank-v3.5` (일부 시도에서 사용)

---

### Part 1. Hybrid + Cohere Rerank 시도

초기 구성은 EnsembleRetriever + ContextualCompressionRetriever(CohereRerank) 조합이었다.
Cohere에 전달하는 문서 내용을 바꿔가며 rerank 품질을 실험하였다.

#### 시도 1-1. Hybrid + Cohere Rerank (full page_content 전달)
- 설정: Vector:BM25 = 6:4, RERANK_TOP=4, BM25 인덱스 전체 내용(subject+text+tables)
- Cohere에 `page_content` 전체(출처년도+subject+text+tables) 전달
- **결과: Cohere Trial key rate limit(10 calls/min) 초과로 크래시 — 결과값 확인 불가**

#### 시도 1-2. Hybrid + Cohere Rerank (subject만 전달)
- rate limit 문제를 우회하기 위해 ContextualCompressionRetriever 대신 커스텀 rerank 함수 구현
- Cohere에 `subject` 필드만 전달하고, retry 로직(최대 3회, 60초 대기) 추가
- **결과: 16/25 (64%)** — Basic RAG(88%)보다 오히려 낮아짐
- **원인 분석**: subject만 전달하면 Cohere가 relevance를 판단할 정보가 너무 부족함. RRF가 이미 잘 정렬한 순서를 오히려 망가뜨리는 역효과 발생

#### 시도 1-3. Hybrid + Cohere Rerank (subject+text 전달)
- Cohere에 `subject + text` 조합 전달 (tables 제외)
- **결과: 20/25 (80%)** — subject만 보낼 때(64%)보다 개선됐지만 Basic RAG(88%)에 미치지 못함
- **원인 분석**: 정보량이 늘어 rerank 품질은 올랐지만, 여전히 RRF 순서를 뒤집어서 성능이 저하됨

#### Part 1 소결
| 방식 | 정답률 |
|------|--------|
| Basic RAG (비교 기준) | 88% |
| Hybrid + Cohere (full text, rate limit 크래시) | 결과 없음 |
| Hybrid + Cohere (subject only) | 64% |
| Hybrid + Cohere (subject+text) | 80% |

→ Cohere Rerank를 사용할수록 성능이 떨어지는 패턴 확인. Reranker에 전달하는 문서 표현과 무관하게 RRF 순서를 뒤집는 것 자체가 문제로 판단하여 Rerank를 제거하는 방향으로 전환.

---

### Part 2. Hybrid only 튜닝 (Rerank 제거 후)

Cohere Rerank를 완전히 제거하고 EnsembleRetriever(RRF)만 사용하면서 파라미터를 튜닝하였다.

#### 시도 2-1. Rerank 제거, 기본 설정 (6:4, top-4, BM25 전체)
- **결과: 20/25 (80%)** — Rerank 제거만으로 64%→80%로 즉시 상승

#### 시도 2-2. 가중치 조정 (6:4 → 8:2)
- Vector:BM25 가중치를 0.8:0.2로 조정
- **결과: 20/25 (80%)** — 변화 없음
- **분석**: 가중치 비율 자체보다 BM25 인덱스에 입력되는 데이터 품질이 문제임을 시사

#### 시도 2-3. BM25 인덱스 입력 데이터 수정 (핵심 개선)
- 기존: BM25 인덱스를 `subject + text + tables` 전체로 구성
- 변경: `subject + text`만 사용, tables 제외
- **결과: 21/25 (84%)** — 개선
- **원인 분석**: 표(tables)는 숫자·코드·표 형식 문자가 많아 TF-IDF 기반 BM25 매칭에서 노이즈로 작용. 제거 후 BM25가 의미 있는 키워드에 집중하게 됨

#### 시도 2-4. RERANK_TOP (LLM에 전달하는 최종 청크 수) 튜닝

| RERANK_TOP | 정답률 |
|------------|--------|
| 4 | 84% |
| 6 | 88% |
| **8** | **92%** ← 최적 |
| 10 | 84% |

- top-8이 스위트스팟: 6보다 많은 컨텍스트로 정답률 상승
- top-10은 오히려 하락 — 노이즈 청크가 LLM 컨텍스트에 끼어들어 핵심 정보를 희석시키는 것으로 판단

### 튜닝 핵심 인사이트

1. **Rerank가 항상 좋은 것은 아니다**: Reranker에 전달하는 문서 표현의 품질과 무관하게, RRF가 이미 잘 정렬한 순서를 뒤집으면 오히려 성능이 떨어질 수 있다.
2. **BM25 입력 데이터 품질이 중요하다**: 표 데이터처럼 통계 기반 키워드 매칭에 노이즈가 되는 데이터를 제거하는 것만으로도 성능이 개선된다.
3. **top-k는 많을수록 좋지 않다**: LLM 컨텍스트에 관련 없는 청크가 많아질수록 핵심 정보가 희석되어 오히려 정답률이 하락한다. 적절한 스위트스팟 탐색이 필요하다.


## Step 3. Basic RAG vs Advanced RAG 비교 분석

### 3-1. 정답률 비교 테이블

| 방식 | 전체 | easy | medium | hard | cross-year | 년도 검색 정확도 |
|------|------|------|--------|------|------------|----------------|
| Basic RAG (벡터 검색만) | 22/25 (88%) | 8/8 (100%) | 6/6 (100%) | 3/4 (75%) | 5/7 (71%) | 25/25 (100%) |
| Advanced RAG (Hybrid RRF) | 23/25 (92%) | 8/8 (100%) | 6/6 (100%) | 3/4 (75%) | 6/7 (86%) | 25/25 (100%) |

### 3-2. 문항별 변화 분석

| 질문 ID | 난이도 | Basic RAG | Advanced RAG | 변화 | 원인 분석 |
|---------|--------|-----------|-------------|------|----------|
| q01~q17 | easy/medium/hard | 정답 (q18 제외) | 정답 (q18 제외) | 유지 | - |
| q18 | hard | 오답 | 오답 | 유지(오답) | 둘 다 LLM 해석 오류. 추나요법 표에서 단순/복잡 구분을 잘못 읽음 |
| q19~q22 | cross-year | 정답 | 정답 | 유지 | - |
| q23 | cross-year | 오답 | **정답** | **개선** | top-8로 더 많은 컨텍스트를 제공하면서 LLM이 올바른 답 생성 |
| q24 | cross-year | 정답 | 정답 | 유지 | - |
| q25 | cross-year | 오답 | 오답 | 유지(오답) | Basic은 검색 실패, Advanced는 검색 성공했으나 LLM 해석 오류로 여전히 오답 |

**변화 요약**
- 개선 1건: q23 (cross-year, 오답→정답)
- 악화 0건
- q25: 오답은 유지됐으나 **검색 자체는 개선됨** (검색 실패 → 검색 성공, LLM 해석 오류로 전환)

### 3-3. 기법별 기여도 분석

> 메타데이터 필터링(`source_year` 기반)은 Basic RAG부터 전 구간에 기적용된 상태이며, 아래 수치는 모두 메타데이터 필터링이 적용된 기준이다.

| 방식 | 정답률 | 비고 |
|------|--------|------|
| Basic RAG (벡터만, 메타데이터 필터링 기적용) | 88% (22/25) | 기준선 |
| + Hybrid Search (6:4, BM25 전체 내용, top-4) | 80% (20/25) | BM25 table 노이즈로 오히려 하락 |
| + Hybrid + Cohere Rerank (subject only 전달) | 64% (16/25) | RRF 순서를 망가뜨림 |
| + Hybrid + Cohere Rerank (subject+text 전달) | 80% (20/25) | 개선됐으나 Basic 미달 |
| + Hybrid 튜닝 (8:2, BM25 subject+text, top-8, Rerank 없음) | **92% (23/25)** | 최종 최적 설정 |

**기법별 실제 기여 요인 정리**

| 기법 | 기여 여부 | 설명 |
|------|----------|------|
| 메타데이터 필터링 | 기여 (기적용) | 년도 정확도 100% 유지의 핵심. 전 구간 적용 |
| BM25 (전체 내용) | 부정적 | table 노이즈로 관련 없는 청크 유입 |
| BM25 (subject+text만) | 긍정적 | 노이즈 제거 후 키워드 매칭 품질 향상 |
| Cohere Rerank | 부정적 | RRF보다 낮은 rerank 품질로 순서 역전 |
| top-k 증가 (4→8) | 긍정적 | LLM에 더 풍부한 컨텍스트 제공, cross-year 문항 개선 |
| top-k 과도한 증가 (→10) | 부정적 | 노이즈 청크 유입으로 핵심 정보 희석 |
