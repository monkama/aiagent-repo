- 예외 케이스 Tool 호출 횟수: `1`

- 예외 케이스 token 사용량: `input 1,214 / output 53 / total 1,267`

## 검색 구조 변경 배경

초기에는 민원 원천 데이터의 `사무명/사무개요`를 청킹 비슷하게 다뤄 벡터 유사도 top-k로 후보를 뽑는 방식을 실험했습니다.

하지만 실제로 써보니 아래 같은 문제가 있었습니다.

- 사용자는 보통 `"퇴직금을 안 줬는데 어디에 신고하나요?"`처럼 넓은 질문을 합니다. 그런데 벡터 검색은 문장 속 단어나 표현이 비슷하다는 이유로 `퇴직연금사업자 등록`, `간이대지급금 지급 청구`처럼 지금 질문과는 결이 다른 민원사무를 위로 올리기도 했습니다.
- 원본 데이터는 질문에 대한 답을 바로 적어둔 FAQ가 아니라, 정부가 관리하는 `민원 사무 목록`입니다. 즉 `"퇴직금을 못 받았을 때 어디에 신고하세요"` 같은 문장이 있는 게 아니라, `근로기준법 위반사실에 대한 신고`, `간이대지급금 지급 청구`처럼 절차 이름과 설명이 들어 있습니다. 그래서 글자나 문장 모양이 비슷하다고 해서 지금 사용자 상황에 맞는 답이라고 보기는 어려웠습니다.
- 특히 사용자는 아직 `처음 신고 단계`를 묻고 있는데, 검색 결과에는 `이미 확인서가 발급된 뒤에 진행하는 절차`나 `더 뒤 단계 신청 업무`가 같이 섞일 수 있었습니다. 예를 들어 `"퇴직금을 안 줬다"`는 질문에 바로 `간이대지급금 지급 청구`가 상위에 뜨면, 사용자가 지금 당장 가야 할 첫 창구를 안내하는 데는 오히려 혼란을 줄 수 있었습니다.

그래서 현재는 `PageIndex`와 비슷한 reasoning-based retrieval 철학으로 바꿨습니다.

- 먼저 `소관부처`
- 그다음 `담당부서`
- 마지막으로 그 범위 안의 `민원사무 record`

를 순서대로 좁혀서 고릅니다.

이렇게 단계별로 나눈 이유는 다음과 같습니다.

- `소관부처 선택`: 전체 검색 공간에서 완전히 무관한 기관을 먼저 제거하기 위해
- `담당부서 선택`: 같은 부처 안에서도 역할이 다른 부서가 많아서 범위를 한 번 더 좁히기 위해
- `레코드 선택`: 마지막에야 `사무명`과 `사무개요`를 읽고 현재 질문과 가장 직접적으로 연결되는 민원사무를 고르기 위해

중간에 `부처+부서`를 한 번에 고르는 구조도 시험했지만, 한 번에 보여줘야 하는 scope 목록이 너무 커져서 input token이 크게 증가했습니다. 실제로 이 실험에서는 정상 케이스 총 token이 `22,891`까지 올라갔고, 그중 `AgencyRoutingTool`만 `21,623`, `scope selection` 입력만 `18,234`를 사용했습니다. 이후 다시 `부처 -> 부서 -> 레코드` 3단계로 되돌리고 payload를 줄인 뒤에는 정상 케이스 총 token이 `7,970`, `AgencyRoutingTool`은 `6,703`까지 내려갔습니다. 그래서 현재는 `부처 -> 부서 -> 레코드` 3단계를 유지한 상태에서 payload를 줄이는 방향으로 최적화를 진행했습니다.

## 비용 병목 분석

비용이 커진 원인:

- 정상 케이스에서 `AgencyRoutingTool`의 입력 컨텍스트가 가장 큰 비용 병목이었습니다.
- 특히 최종 민원사무 선택 단계에서 `candidate_records` 전체를 너무 두껍게 LLM에 전달하고 있었습니다.

근거:

- 정상 baseline의 전체 input token `12,302` 중 `AgencyRoutingTool`이 `11,087`을 사용했습니다.
- 즉 전체 input token의 약 `90%`가 라우팅 단계에서 발생했습니다.
- `candidate_records` 수는 `14개`였고, 각 record마다 아래 필드를 모두 넣고 있었습니다.
  - `record_id`
  - `scope_key`
  - `분류번호`
  - `사무명`
  - `사무개요`
  - `소관부처`
  - `담당부서`
  - `접수방법`
  - `접수/처리`
  - `근거법령`
  - `구비서류`

## 적용한 최적화

선택한 최적화:

- `Context pruning`

선택 이유:

- 현재 구조에서 가장 큰 비용은 `AgencyRoutingTool`의 retrieval observation 크기였습니다.
- Tool 호출 횟수나 모델을 바꾸기 전에, 라우팅 판단에 꼭 필요하지 않은 필드를 줄이는 것이 가장 안전한 1차 최적화라고 판단했습니다.

변경 내용:

- `AgencyRoutingTool`에서 LLM에 전달하는 관찰 payload를 전체적으로 축소했습니다.
- record 당 전달 필드를 `11개 -> 7개`로 줄였습니다.
- 제거한 필드:
  - `접수방법`
  - `접수/처리`
  - `근거법령`
  - `구비서류`
- `사무개요` 길이도 `500자 -> 320자`로 줄였습니다.
- ministry selection 단계에서는 부처 후보마다 `ministry_name`만 남기고 `department_count`, `record_count`를 제거했습니다.
- department selection 단계에서는 scope 후보마다 `scope_key`만 남기고 `ministry_name`, `department_name`, `record_count`를 제거했습니다.
- 유지한 필드:
  - `record_id`
  - `scope_key`
  - `분류번호`
  - `사무명`
  - `사무개요`
  - `소관부처`
  - `담당부서`

## Before / After 비교

정상 케이스 비교:

| 항목 | Before | After | 변화 |
|------|--------|-------|------|
| LLM 호출 횟수 | 4 | 4 | 동일 |
| Input token | 12,302 | 7,597 | `-4,705` |
| Output token | 328 | 371 | `+43` |
| Latency 또는 전체 실행 시간 | 15,355 ms | 19,323 ms | `+3,968 ms` |
| Tool 호출 횟수 | 2 | 2 | 동일 |
| Retrieval context 수 | 14 records | 14 records | 개수 동일, record 당 필드 `11 -> 7`, 부처/부서 payload도 축소 |

예외 케이스 비교:

| 항목 | Before | After | 변화 |
|------|--------|-------|------|
| LLM 호출 횟수 | 1 | 1 | 동일 |
| Input token | 1,214 | 1,214 | 동일 |

| Latency 또는 전체 실행 시간 | 12,515 ms | 6,423 ms | `-6,092 ms` |
| Tool 호출 횟수 | 1 | 1 | 동일 |

추가 관찰:

- 정상 케이스에서 `AgencyRoutingTool` input token은 `11,087 -> 6,382`로 줄었습니다.
- 총 token도 `12,630 -> 7,970`으로 줄었습니다.
- 즉 이번 최적화는 의도한 대로 `라우팅 입력 token`을 직접 줄였고, 전체 실행 비용도 함께 낮췄습니다.
- 반면 latency는 오히려 늘었는데, 이번 최적화의 목표는 latency가 아니라 `input token 감소`였고, 현재 구조에서는 네트워크/모델 응답 변동성이 더 크게 작용한 것으로 보입니다.

## 동작 유지 확인

6~8주차에서 사용한 성공 기준 중 이번 비교에 사용할 항목:

- 정상 케이스에서 소관부처와 민원사무가 크게 어긋나지 않는가
- 지역 정보가 필요한 경우 추가 질문으로 빠지는가
- 예외 입력에서 무리하게 잘못된 기관을 확정하지 않는가

최적화 전후에 동일하게 유지된 동작:

- 정상 케이스에서 여전히 `고용노동부 -> 근로기준정책과 -> 근로기준법 위반사실에 대한 신고` 흐름으로 라우팅되었습니다.
- 예외 케이스에서는 여전히 위치 정보를 다시 묻고, 답변이 없으면 `follow_up_abandoned`로 종료되었습니다.
- 최종 답변에는 여전히 `관련 법령`, `구비 서류`가 포함되었습니다. 이 정보는 라우팅용 LLM 입력에서는 제거했지만, 최종 레코드가 선택된 뒤 코드 후처리로 다시 붙도록 유지했습니다.

달라진 동작:

- 최종 자연어 답변 문구는 조금 달라졌습니다.
- output token과 latency는 줄지 않았고, 이번 측정에서는 오히려 소폭 증가했습니다.
- 이번 최적화는 `정답을 바꾸기`보다 `LLM이 판단할 때 보는 입력 크기를 줄이기`에 초점을 둔 변경이었습니다.

문제가 된다면 되돌릴 변경:

- 만약 이후 테스트에서 라우팅 품질이 떨어지면, `사무개요` 길이 제한을 다시 늘리거나 일부 필드만 선택적으로 복구할 수 있습니다.
- 예를 들어 `record selection` 단계에서 현재 제거한 필드 중 일부를 다시 넣어야 할 수도 있습니다. 다만 지금은 `관련 법령`과 `구비 서류`를 LLM 입력에 포함하지 않아도 최종 답변 품질이 유지되고 있어, 우선은 이 구조를 유지하는 편이 낫다고 판단했습니다.

## 다음 최적화 계획

다음에 시도할 최적화:

- `AgencyRoutingTool`의 `candidate_records` 범위를 주제별로 나누어 LLM이 봐야할 레코드 수 축소: 민원사무를 주제별로 한 번 더 묶고, LLM이 필요한 묶음만 읽게 하는 방식 검토


이유:

- 이번에는 가장 큰 병목이었던 `record selection` observation과 부처/부서 선택 payload를 함께 줄였습니다.
- 현재도 정상 케이스 기준 `AgencyRoutingTool` input token이 `6,382`로 가장 크기 때문에, 다음에는 `무조건 후보 개수를 줄이기`보다 `후보를 더 잘 묶어서 필요한 범위만 읽게 하는 방식`과 `record selection` 단계 필드 추가 축소를 함께 검토하는 것이 자연스러운 다음 순서라고 판단했습니다.
